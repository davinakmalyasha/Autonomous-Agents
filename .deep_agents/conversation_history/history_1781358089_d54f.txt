# Goal
Implement a custom priority queue with aging to prevent starvation

## Codebase Analysis & Root Cause
The workspace is empty. The user wants a priority queue (classes with different importance) with an aging mechanism to prevent low-priority items from starving indefinitely. Common approaches:
1. **Sorted list / heapq**: Simple, O(log n) push/pop, but aging requires periodically boosting priorities of waiting items.
2. **Heap with aging boost**: Use a min-heap of (priority, timestamp, item). Aging can be implemented by decreasing priority over time (e.g., periodically increase priority of items). However, modifying priority in a heap is not straightforward; we can store an 'entry_time' and compute dynamic priority using 'age = time_elapsed', then when popping, compute effective priority = base_priority - age * aging_factor. This avoids in-place modifications but could have stale entries. Alternatively, use a custom class that tracks creation time and updates priority on each pop.
3. **Multiple heaps (multi-level queue)**: Separate heaps per priority tier, with weight-based selection and aging by promoting items to higher tier after timeout.
4. **Balanced BST / skip list**: O(log n) operations and easy to find min, but more complex to implement.

## Proposed Changes
### Chosen Design: List-based priority queue with aging
- **Data Structure**: Sorted list (using `bisect.insort`) ordered by base priority.
- **Aging**: When popping, compute effective priority = `base_priority - aging_factor * (current_time - entry_time)` for all items and select the minimal one.
- **Trade-offs**:
  - Push: O(n) (insertion into sorted list)
  - Pop: O(n) (scan all items for effective priority)
  - Simple, correct, good for moderate queue sizes.
- **Alternatives considered**:
  1. **Heap with lazy reordering** (heapq) – O(log n) push/pop but may not perfectly handle dynamic aging without periodic re-heapify. Documented as alternative.
  2. **Multi-level feedback queue** – multiple heaps per priority band with promotion timers – more efficient for large queues.
  3. **Balanced BST / skip list** – O(log n) for all operations but complex to implement from scratch.
